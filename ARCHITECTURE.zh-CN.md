# 设计文档

[English](./ARCHITECTURE.md) | [简体中文](./ARCHITECTURE.zh-CN.md)

本文档是架构与设计决策的唯一事实来源。

## 范围

- AI 优先的 DSL 建模流程
- 渲染层与几何计算层解耦
- 面向多用户、多项目版本管理的数据库持久化
- 面向渲染与制造场景的导出管线

## 核心原则

- Three.js 是渲染与交互层，不是几何真值来源。
- DSL 是语义化场景描述，不是网格文件格式。
- 通过 schema 与解释器边界隔离变更，降低重构成本。
- 几何精度采用模式化策略：默认快，按需准。
- 持久化层必须支持归属、历史与可审计性。

## 分层架构

| 层次 | 输入 | 输出 | 数据形态 | 主要职责 | 目标 |
|---|---|---|---|---|---|
| AI 生成层 | 自然语言 prompt + 可选当前场景上下文 | 候选 DSL 载荷 | JSON 对象 | 将用户意图转换为结构化场景意图 | 稳定且可编辑的建模输入 |
| DSL Schema 层 | 原始 DSL 候选 | 校验并归一化后的 DSL | JSON 对象 | 校验、默认值补齐、约束与兼容处理 | 可预测且安全的 DSL 契约 |
| 持久化层（数据库） | 项目事件、DSL 快照、元数据、制品引用 | 可追溯项目与版本记录 | 数据库记录 + 对象存储引用 | 多用户隔离、项目归属、版本历史、分支、审计 | 支持长期协作与演进 |
| 交付层（API） | 查询条件、命令、ID | 类型化响应、流与二进制 | JSON/HTTP + 二进制载荷 | 鉴权、授权、版本查询、patch 应用、导出编排 | 提供稳定的产品接口契约 |
| DSL 解释器层 | 来自持久化/API 的归一化 DSL | Geometry IR（几何中间表示） | 内存对象 | 将语义节点映射为几何命令和变换 | 解耦建模语义与底层引擎实现 |
| 几何计算层（可选） | Geometry IR | 精度受控的几何结果 | 内存对象（mesh 或 B-Rep） | 布尔运算、参数求解、公差处理 | 按需提供高保真建模能力 |
| 渲染层（Three.js） | 可渲染 mesh/scene graph | 交互视口帧 | 运行时 GPU 对象 | 预览、镜头控制、操作反馈 | 快速可视化迭代 |
| 导出层 | 几何结果 + 元数据 | 交换/制造格式文件 | `.glb`, `.stl`, `.step`, `.3mf`, `.zip` | 格式转换、打包、兼容性检查 | 打通设计到 CAD/制造链路 |

## 几何计算策略

### 默认：低保真模式

- 解释器直接输出可渲染 mesh。
- 优先保证交互响应与迭代速度。
- 适合概念设计、结构布局与快速验证。

### 可选：高保真模式

- 解释器按策略将选定节点送入几何计算层。
- 在渲染或导出前应用精度与公差规则。
- 适合制造前准备与严格尺寸要求。

## 持久化策略（数据库优先）

- 用户可以拥有多个项目，且具备严格访问隔离。
- 每个项目维护版本时间线（快照 + patch 历史）。
- DSL 快照与几何制品支持独立版本化。
- 大文件导出通过制品记录关联对象存储。
- API 支持回滚、对比与分支/合并式迭代。

## 为什么这个架构可扩展

- DSL 契约变更局部收敛在 schema 与解释器。
- 几何引擎升级不会强制重写渲染层。
- Three.js 可作为可替换的表现层组件。
- 存储实现可演进而不破坏 DSL 语义。

## 实施建议

- 让 Geometry IR 显式且带版本号。
- 将各导出器实现为 Geometry IR 适配器。
- 高保真计算与导出使用异步任务编排。
- 尽量保证 patch 操作幂等。
